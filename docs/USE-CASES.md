# Use Cases

StellaAgent를 어떤 상황에서, 어떻게 사용하는가.

---

## 1. 새 프로젝트 온보딩

**상황**: 팀에 합류했거나, 오픈소스 프로젝트를 처음 본다. README와 폴더 구조만으로는 전체 그림이 그려지지 않는다.

**StellaAgent로 하는 일**:

```bash
STELLA_TARGET=/path/to/new-project npm run dev
```

브라우저를 열면 프로젝트의 성좌가 보인다. 큰 별(복잡한 파일)이 어디에 있는지, 어떤 파일들이 밀집해 있는지(강한 의존성), 어떤 파일이 외곽에 떨어져 있는지(독립적) — 30초 안에 전체 구조가 파악된다.

**핵심 가치**: 코드를 읽기 전에, 먼저 관측한다.

---

## 2. 숨겨진 의존성 발견

**상황**: `config.ts`를 수정했는데 예상치 못한 곳에서 버그가 터진다. import 관계로는 연결이 없는데 왜?

**StellaAgent로 하는 일**:

GitPanel에서 co-change 분석을 확인한다. `config.ts`와 `auth.ts`가 75% 커플링 — import는 없지만 git 기록상 항상 같이 바뀐다. 이것은 숨겨진 의존성이다. teal 색 성좌선이 이 관계를 시각적으로 보여준다.

**핵심 가치**: import에 없는 의존성을 git에서 발견한다.

---

## 3. AI 에이전트 감사

**상황**: Claude Code와 Copilot을 적극 사용하고 있다. AI가 어디를 얼마나 바꿨는지 파악하고 싶다.

**StellaAgent로 하는 일**:

AgentPanel에서 에이전트별 활동을 확인한다. Claude Code가 주로 `server/` 영역을 수정하고, Copilot은 `client/` 영역에서 활동한 것이 trail로 보인다. 에이전트가 수정한 파일이 hot file과 겹치면 — 그 부분은 특히 주의 깊게 리뷰해야 한다.

**핵심 가치**: AI 협업의 투명성. 추적이 아니라, 관측.

---

## 4. 리팩토링 타겟 선정

**상황**: 기술 부채를 줄이고 싶은데, 어디부터 손대야 할지 모르겠다.

**StellaAgent로 하는 일**:

두 가지 지표를 조합한다:
- **Hot files** (자주 바뀌는 파일) = 변경 빈도가 높다
- **High co-change coupling** (여러 파일과 높은 커플링) = 수정 시 파급 범위가 넓다

이 두 조건을 동시에 만족하는 파일이 리팩토링 1순위다. 성좌에서 크고, 많은 선이 연결된 별이 바로 그것이다.

**핵심 가치**: 감이 아니라 데이터로 우선순위를 정한다.

---

## 5. 코드 리뷰 보조

**상황**: PR이 올라왔다. 변경된 파일 목록은 있지만, 이 파일들이 프로젝트에서 어떤 위치에 있는지 맥락이 없다.

**StellaAgent로 하는 일**:

SearchBar에서 PR에 포함된 파일을 검색한다. 성좌에서 해당 별들이 하이라이트된다. 이 별들이 한 군집에 있으면 — 변경의 범위가 좁고 안전하다. 여러 군집에 흩어져 있으면 — 여러 모듈에 걸친 변경이므로 주의가 필요하다.

co-change 분석과 대조하면, PR에 빠진 파일(같이 바뀌어야 하는데 빠진 파일)도 발견할 수 있다.

**핵심 가치**: 파일 목록이 아니라, 구조적 맥락에서 리뷰한다.
